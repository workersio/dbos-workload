# E-016 Committed Sys-DB Retry Re-Entry Idempotence

## Product Promise

DBOS system database retry loops preserve exactly-once durable semantics when a
connection failure causes retry re-entry after a write may already have
committed. Message receives, child workflow recording, and implicit child
`get_result` checkpoints must not duplicate effects, skip checkpoint slots,
consume additional messages, or misclassify idempotent replays as
nondeterministic workflow conflicts.

## Fault Dimensions

### committed-sysdb-retry-reentry

Purpose: verify that sys-db methods wrapped in retry logic are safe when a
retriable database connection failure causes the same logical operation to run
again after durable state may already exist.

Trigger point: after the workload has durable evidence for the logical
operation row, consumed message, or child edge, it re-enters the same workflow
ID and function ID through a deterministic retry injection or an equivalent
explicit replay surrogate.

Injection: use a scoped engine/proxy wrapper around `SystemDatabase.engine` for
the selected case when feasible, or explicitly perform the second logical call
after proving the first durable state exists. The run artifact must say which
method was used and why it preserves the committed-state risk.

Reachability evidence: PR `#740` added focused target tests for
`recv_consume`, `record_child_workflow`, and `record_get_result`; target
`dbos/_sys_db.py` now captures `record_get_result` ids outside retry,
checks recorded `recv_consume` output before consuming another message, and
treats same-child unique conflicts as idempotent.

Expected product behavior: replaying the same logical operation returns or
preserves the already-recorded effect; it must not consume a second message,
advance `function_id` twice, duplicate operation rows, or conflict on the same
child edge.

Oracle impact: the independent operation ledger and read-only system table
inspection fail if DBOS duplicates, drops, or misclassifies committed retry
state.

Replay metadata: expected command shape
`python .workers/workloads/system-db-retry-idempotence/system_db_retry_idempotence_workload.py --rung rung-001-committed-sysdb-retry-reentry --case <case-id> --seed <seed>`.

## Oracle

The workload should build an independent ledger keyed by workflow ID, function
ID, operation name, and case ID. It must compare the ledger with public handles
and read-only DBOS rows:

- `operation_outputs` row count, function ID, function name, output/error, and
  child workflow ID.
- `notifications` consumed flags and payloads for receive cases.
- `workflow_status` rows for parent and child handles where applicable.
- Public handle result/error observations after bounded polling or retrieval.

Required invariant checks:

- receive replay returns the recorded output and does not consume a later or
  second message;
- no duplicate operation row appears for a single logical function ID;
- same child edge replay is idempotent, different child at the same function ID
  conflicts, and empty child IDs are rejected before durable write;
- implicit `DBOS.getResult` retry advances parent `function_id` exactly once
  and records one child reference row;
- retry injection or replay surrogate is proven in the artifact before any
  case can pass.
