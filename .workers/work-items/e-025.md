## Fault Details

### client-get-event-prompt-polling

Purpose: verify PR `#713`'s client `get_event` polling fix beyond one sync product test.

Fault model: `DBOSClient` has no notification listener, so a waiting client can only observe a later `set_event` through database re-checks. A regression can wait until the full timeout, return stale event values, miss terminal workflow state, or leak waiter map entries.

Oracle impact: public client result, latency, durable event row, workflow terminal status, and `workflow_events_map` cleanup must agree with the modeled event schedule.

## Parameter Matrix

| Case | Seed | Schedule | Fault Model | Expected Focus |
|---|---:|---|---|---|
| `case-001` | 7130 | sync client delayed event | workflow sets an event after a modeled delay while sync `DBOSClient.get_event` waits with a long timeout | client returns the modeled value within the prompt polling bound, not near the full timeout; event row and handle result agree |
| `case-002` | 7131 | async client delayed event | same as case 001 through `DBOSClient.get_event_async` with concurrent async workflow execution | async client polls without blocking the event loop, returns promptly, and leaves no waiter entry |
| `case-003` | 7132 | client terminal missing event | workflow reaches terminal success without setting the requested key while client waits | client returns `None` within the terminal/miss bound instead of waiting for the full timeout, and terminal status is unchanged |
| `case-004` | 7133 | client event update race | workflow sets an initial value, later updates the same key, while client calls before and after the update window | first client read observes the initial modeled value promptly; later read observes the updated modeled value; no stale waiter or old value leaks across reads |

## Oracle

- Persist workflow IDs, event keys, client path, timeout, event-set delay, polling interval, observed latency, final event table value, waiter payload, workflow terminal status, and cleanup observations.
- Check that sync and async client waits finish within the modeled prompt bound derived from `_event_recheck_interval()` plus scheduling slack.
- Fail on delayed delivery until the full timeout, wrong or stale event value, async client blocking the event loop, missing-key wait hanging past terminal state, leaked waiter payload, or terminal row mutation.

## Coverage Gap

Existing message/event workloads already cover runtime `get_event` fallback when the listener is stopped and client stream reader fallback/resume. They do not cover DBOSClient `get_event` no-listener polling, which is the specific surface fixed by PR `#713`.

Cloud evidence found that `DBOSClient.get_event` returns event values promptly

client timeout before returning `None`. The failing invariant is
`client_terminal_miss_within_prompt_bound`.
