## Fault Details

### custom-decorator-entrypoint-matrix

Purpose: verify the API contract implied by PR `#706` across decorator order, sync/async parity, public entry points, failures, replay, recovery, and class metadata.

Fault model: DBOS metadata is attached to one callable while application code invokes or enqueues another wrapper in the `__wrapped__` chain. Public entry points may resolve the wrong name, durable rows may store wrapper names, or replay/recovery may rerun hooks incorrectly.

Oracle impact: hook ledger, public result/error, durable workflow status, operation output names, class metadata, and replay/recovery call counts must agree with the modeled decorator composition.

## Oracle

- Persist workflow IDs, DBOS registered names, wrapper order, hook ledger events, entry point used, expected step/function names, durable status rows, and replay/recovery observations.
- Check that `workflow_status.name`, step/function names, list/status API observations, public handle results/errors, and hook counts all match the model.
- Fail on workflow-not-found from a well-formed wrapper, wrong durable name, missing class metadata, duplicate hook or transaction side effect on replay, recovery using an unwrapped function path, or application error masked by wrapper metadata.

## Coverage Gap

PR `#706` covers one async happy path with `@DBOS.workflow` outside one custom decorator. Existing workload areas cover async checkpointing, serialization, queues, and lifecycle behavior, but none model ordinary application decorators as the user/API composition boundary. This rung adds decorator order, durable metadata, failure/replay, recovery, and class-method oracles.
