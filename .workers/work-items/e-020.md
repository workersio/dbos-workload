# E-020 Temporal Workflow Introspection

## Product Promise

Workflow temporal introspection exposes durable completion/dequeue timestamps
consistently across runtime, client, conductor, queued/direct workflows,
terminal transitions, resume, filtering, aggregation, and export/import.
`completed_at` must be set for terminal workflows, cleared on resume, and used
by `completed_after` / `completed_before` filters. `dequeued_at` must reflect
queue start time and drive `dequeued_after` / `dequeued_before` filters without
matching direct workflows. Operation-output timing must remain monotonic and
usable for step-list, time-bucket, selectable aggregate, and latency APIs.

## Fault Dimensions

### temporal-window-filter-drift

Purpose: prove public workflow temporal filters match the independent model
across direct success/error, queued success, delayed queue start, cancellation,
resume, and relaunch.

Trigger point: create modeled workflows around recorded wall-clock windows,
capture public status snapshots before and after each transition, then query
through runtime and client APIs with `completed_after`, `completed_before`,
`dequeued_after`, and `dequeued_before`.

Injection: vary terminal state, queue/delay path, resume after cancellation,
manual/direct execution, public API surface, and restart boundary.

Reachability evidence: PR `#681` added `completed_at` and completion filters;
PR `#682` moved workflow terminal/resume timestamps to database time; PR `#674`
added aggregate time buckets; PR `#685` added selectable aggregate outputs and
latency maxes. Target tests cover narrow examples, but do not compose terminal
transitions, queue dequeue windows, relaunch, client parity, conductor parity,
aggregate selections, and model intersections in one workload.

Expected product behavior: temporal filters return exactly the workflow IDs in
the independent model; pending/resumed-not-yet-terminal workflows do not match
completion windows; direct workflows do not match dequeue windows; queued rows
match dequeue windows only after actual worker start.

Oracle impact: the workload fails if a timestamp is missing, uses application
clock drift inconsistently, survives resume incorrectly, is not preserved across
relaunch/import, or leaks/misses rows in a composed temporal query.

### terminal-resume-timestamp-transition

Purpose: prove terminal-state timestamp semantics survive cancellation and
resume without stale completion markers.

Trigger point: cancel a blocked workflow, observe `completed_at`, resume it,
observe `completed_at=None` before release, then release and verify a fresh
terminal timestamp.

Injection: run both runtime and client status/list paths after each state
transition and after process relaunch.

Expected product behavior: `CANCELLED` has a completion timestamp, resumed
active state clears it, final terminal state receives a new timestamp, and no
completed-window query returns the resumed workflow while it is active.

### aggregate-step-time-bucket-drift

Purpose: prove workflow aggregate latency, selectable aggregate output, time
bucket, and step timing APIs use the same durable temporal model as workflow
listing.

Trigger point: run direct and queued workflows with sync/async steps, blocking
durations, and known order; then compare `list_workflow_steps`,
`get_workflow_aggregates`, conductor aggregate request/response payloads, and
workflow status timestamps.

Injection: include step rows with completed operation outputs plus bookkeeping
rows that should not appear in completed operation-output windows.

Expected product behavior: step timestamps are monotonic, time bucket values
match the modeled bucket boundaries, `select_count` / `select_min_created_at`
outputs are present only when requested, aggregate `max_queue_wait_ms` is null
for direct-only groups and non-negative for queued groups, aggregate
`max_total_latency_ms` is non-negative for terminal groups, and
operation-output completed windows exclude bookkeeping rows with null
completion time.

## Oracle

The workload should build an independent temporal ledger with:

- seed, derived workflow IDs, case windows, and target commit;
- public transition observations for created, dequeued, cancelled, resumed,
  completed, exported, imported, and relaunched states;
- expected membership for every completed/dequeued window and composed filter;
- runtime and client query results for the same filters;
- read-only workflow status rows and operation-output timing rows when public
  APIs do not expose enough detail;
- aggregate results grouped by status/name/time bucket where relevant, including
  selected count, minimum created-at, queue-wait, and total-latency outputs.

Required invariant checks:

- `created_at <= completed_at` for terminal rows and `completed_at is None` for
  active/resumed rows;
- cancelling a nonterminal workflow sets `completed_at`, resuming clears it,
  and final completion sets a fresh timestamp;
- `completed_after` / `completed_before` and `dequeued_after` /
  `dequeued_before` filters equal the independent model across runtime and
  client APIs;
- direct workflows never appear in dequeue-window queries;
- queued workflows have `dequeued_at >= created_at` and delayed workflows have
  `dequeued_at >= delay_until_epoch_ms`;
- export/import and relaunch do not recompute or drop temporal fields;
- operation-output step timing is monotonic, aggregate time buckets match the
  independent model, selected aggregate outputs are present/absent as
  requested, aggregate queue/total latency is non-negative where defined, and
  completed operation-output windows do not include bookkeeping rows with null
  completion time.
