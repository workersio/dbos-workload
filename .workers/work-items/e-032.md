# E-032 DBOS.send From a Step Is Not Exactly-Once

## Product Promise

`DBOS.send` / `DBOS.send_bulk` is a durable messaging primitive; a recipient using
`DBOS.recv` receives each logical message once. The public API draws no distinction
between sending from a workflow and sending from a step.

## Mechanism (source-grounded, live on release AND main)

`send_bulk` dispatch (`dbos/_core.py:2079`): `if cur_ctx.is_workflow()` records the
send as a guarded step (`workflow_id` + `function_id`). The `else` branch handles
BOTH client and step contexts and passes `workflow_id=None, function_id=None`. In
`_send_bulk_txn` the OAOO block is gated on `workflow_id is not None`
(`_sys_db.py:2818/2890`), so it is skipped for the step path; with no
`idempotency_key`, each call mints a fresh `message_uuid` (`_sys_db.py:2852`) and
the `on_conflict_do_nothing` (`:2876`) never dedups. A `@DBOS.step(max_attempts>1)`
re-runs its body under the same step on each retry, so a step that sends then fails
re-inserts the message every attempt. Exact structural analog of e-031
(`write_stream` from a step) on the `notifications` table / `DBOS.send` API, and of
e-028 (a recorded-operation guard present on one sibling path, absent on the other).

## Verdict

the recipient; the same send from a workflow context delivered it **once**. Novel,
