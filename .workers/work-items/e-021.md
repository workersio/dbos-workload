# E-021 Preemptible Async Step Cancellation

## Product Promise

Preemptible async steps cancel promptly when their workflow is cancelled,
without waiting for the application-level await to finish. A cancellation that
preempts a running step must not record a durable step error or output, must not
enter the ordinary retry loop, and must let a later workflow resume rerun the
step from scratch and record exactly one successful step result. The same
promise must hold when preemption is configured through `@DBOS.step(...)` or
through `DBOS.run_step_async(...)` step options.

## Fault Dimensions

### preempted-step-checkpoint-poisoning

Purpose: prove that cancelling a preemptible step does not record a durable
operation-output error that later resume treats as the step result.

Trigger point: start an async workflow whose first invocation enters a
preemptible step and blocks on a long await or private application gate. Cancel
the workflow while the step is inside that await, before the gate is released.

Injection: use `DBOS.cancel_workflow_async` or `DBOS.cancel_workflow` after the
step-start ledger is durable enough for the workload to know the step is
running. Keep the first invocation blocked unless DBOS preemption cancels it.

Expected product behavior: the first invocation observes `asyncio.CancelledError`
inside the application step, the public handle reports workflow cancellation,
`list_workflow_steps` shows no output/error row for the preempted attempt, and
`resume_workflow_async` reruns the step and records exactly one successful
output.

Oracle impact: the workload fails if the preempted attempt records an error
row, resume replays the cancelled attempt, the final step list contains
duplicate rows, or the successful resumed output is missing.

### preemptible-cancellation-retry-storm

Purpose: verify that preemption bypasses ordinary step retry machinery even
when the step is configured with retries and retry validators.

Trigger point: configure the preemptible step with `retries_allowed=True`,
multiple `max_attempts`, short retry intervals, and an optional `should_retry`
ledger.

Injection: cancel the workflow while the first step attempt is pending.

Expected product behavior: cancellation produces one application step
invocation before resume, no retry sleeps or retry validator calls for the
preempted cancellation, and no `DBOSMaxStepRetriesExceeded` terminal error.

Oracle impact: invocation count, retry ledger entries, elapsed bounded window,
and durable status/error fail if cancellation is routed through normal retry
handling.

### preemptible-option-path-drift

Purpose: verify that the `preemptible=True` contract is consistent across the
decorator path and `DBOS.run_step_async` step-options path, and that the option
meaningfully differs from a non-preemptible long step.

Trigger point: run matched workflows using decorated preemptible steps,
`DBOS.run_step_async({"preemptible": True}, ...)`, and a non-preemptible
control that blocks on the same application gate.

Injection: cancel each workflow after its long step starts but before the gate
is released.

Expected product behavior: preemptible variants observe application cancellation
and settle without releasing the gate; the non-preemptible control does not
claim the application step was interrupted before the gate is released.

Oracle impact: the workload fails if the options path does not preempt, if the
decorator path and options path disagree on durable outcomes, or if the
preemptible flag has no observable difference from the non-preemptible control.

## Oracle

The workload should build a preemption ledger with:

- rung/case/seed and target commit;
- workflow IDs, configuration path (`decorator` or `run_step_async`), retry
  options, and whether the case uses a non-preemptible control;
- application step invocation counts, cancellation observations, gate release
  order, retry validator calls, and elapsed bounded windows;
- public handle result/error and durable workflow status before cancel, after
  cancel, after resume, and after terminal completion;
- `list_workflow_steps` output for the workflow before resume when possible and
  after final completion;
- read-only operation-output row details if public step listing cannot
  distinguish a missing preempted row from a recorded null output.

Required invariant checks:

- every case reaches the modeled in-step blocked state before cancellation is
  injected;
- preemptible application code observes cancellation without the workload
  releasing the application gate first;
- preempted cancellation produces exactly one application invocation before
  resume and does not call retry validators or exhaust retries;
- before resume, the preempted attempt has no durable step output/error row;
- after resume, the final workflow returns the modeled result and exactly one
  successful step output row exists for the logical step;
- decorator and `run_step_async` option paths agree on cancellation, resume,
  and operation-output isolation;
- the non-preemptible control does not falsely report application interruption
  before gate release;
- no poller or step task leak is visible after terminal observation.
