# E-015 SQL Auth Denial Leaves Queued Workflow Pending

## Product Promise

helper must carry persisted authentication metadata through dequeue,
required-role checks, runtime/client/direct-SQL observations, recovery, and
terminal status updates.

## Fault Dimensions

### sql-origin-required-role-denial

Purpose: verify that a SQL-enqueued workflow with persisted roles that fail a
Python `@DBOS.required_roles(["admin"])` check reaches a terminal authorization
error instead of staying pending.

Trigger point: queue worker dequeues a SQL-origin row with user `bob` and roles
`["reader"]` for a workflow requiring `admin`.

Injection: insert queued workflow rows through the SQL helper so auth metadata
comes from persisted SQL state rather than Python middleware context.

Reachability evidence: WIO run `01KVVWWET7CJJR9BCMGDQ2QB7Q` completed the
allowed-role control and then left denied `case-002` pending; focused
confirmation `01KVVX0D0XB2H6AHBCMD0GKD4X` reproduced the denied case.

Expected product behavior: denied execution should finalize as authorization
`ERROR` with persisted error/output semantics visible to runtime, client, and
direct SQL observers.

Oracle impact: a dequeued denied row that remains `PENDING` with null
`output`/`error` after the oracle timeout is a liveness and terminal-state
violation.

Replay metadata: primary run `01KVVWWET7CJJR9BCMGDQ2QB7Q`, confirmation
`01KVVX0D0XB2H6AHBCMD0GKD4X`, harness commit
`4b9c682485b897351279b0f0366c87feb76e7ac3`.

## Workload Plan

The historical workload uses real Postgres, default build profile, and
`.workers/workloads/auth-context-sql-enqueue/auth_context_sql_enqueue_workload.py`.
Controls must include an allowed SQL-origin required-role case so setup,
migration, queue registration, and persisted auth metadata are proven before
classifying a denial failure.

## Oracle

Allowed SQL-origin auth rows must succeed and match runtime, client, direct SQL,
list, and body-result observations. Denied SQL-origin auth rows must reach
terminal `ERROR`; they must not remain `PENDING` after dequeue.

## Execution / Evidence Notes

The primary run passed allowed-role controls for user `alice` with roles
`["reader", "admin"]`. The denied user `bob` with roles `["reader"]` remained
`PENDING` with null `output` and `error` after the oracle timeout. The focused
confirmation reproduced the denied case without the allowed control.
