## Fault Details

### rate-limit-partial-index-plan

Purpose: verify the production performance bug from issue `#696`, where correct rate-limit behavior could still cause high CPU because Postgres did not use the `idx_workflow_status_rate_limited` partial index.

Fault model: populate the DBOS system `workflow_status` table with many unrelated rows, many old rate-limited rows, and a small recent window for one queue. A regression can compile the rate-limit predicate in a way that misses the partial index, fall back to a full table scan, or return the wrong recent-start count.

Oracle impact: the workload must record the exact query, `EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)`, index definitions, modeled/observed recent count, and functional limiter sanity evidence.

## Oracle

- Check the Postgres schema has `idx_workflow_status_rate_limited` with a predicate equivalent to `rate_limited = TRUE`.
- Execute the compiled rate-limit count query shape from `target/dbos/_sys_db.py` against populated Postgres data.
- Fail if the `EXPLAIN` JSON shows a full sequential scan of `workflow_status` instead of `idx_workflow_status_rate_limited` or an explicitly recorded equivalent partial index at the modeled row volume.
- Check the query returns the modeled recent-start count for the selected queue/partition.
- For `case-003`, also check live limiter behavior: no extra start when the modeled recent count is at the limit, and progress after the window advances.
- Record latency and buffer counts as diagnostics, but do not make noisy absolute timing the only oracle.

## Coverage Gap

Existing queue rungs cover functional rate-limit windows, live limiter changes, partition behavior, result retrieval, and cleanup. Issue `#696` was a query-plan regression: behavior could pass while the scheduler query ignored the partial index and burned CPU at production row counts. This rung adds a Postgres plan/selectivity oracle rather than a new seed sweep.
