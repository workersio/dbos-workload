# E-019 Queued Async Task Retention Under GC

## Product Promise

Queued async workflows remain strongly reachable after queue dequeue and handle
conversion to polling handles. Cyclic GC must not destroy a pending async
workflow task while it is suspended, and DBOS must release the task reference
after completion, cancellation, or modeled application error.

## Freshness / Novelty Check

- Target ref: `3df88c4bcc3a2b73d91b765b459f4c7beb6a690c`.
- Related issue/PR: issue `#710` and PR `#711` "Keep a strong reference to
  running async workflow tasks so GC cannot destroy them mid-execution".
- Freshness classification: `current-regression-guard`; the current target
  includes PR `#711`, so this is a green/regression candidate, not an expected
  current-target bug.
- Novelty classification: existing `E-004` green evidence covers async gather

## Fault Dimensions

### queued-async-task-gc-destruction

Purpose: verify that queued async workflows cannot be garbage-collected while
pending after `execute_workflow_by_id(...)` has discarded the original async
task handle and returned a polling handle.

Trigger point: after queue worker dequeue, after the async workflow reaches a
known suspended await on a future rooted only in its coroutine frame, and before
the workload unblocks that future.

Injection: expose only weak references to each pending future, force repeated
`gc.collect()` cycles with bounded allocation pressure, and retain only public
workflow handles/status APIs in the harness.

Reachability evidence: issue `#710` and PR `#711` show that without
`dbos._workflow_tasks`, cyclic GC can destroy the pending task and record
`RuntimeError: coroutine ignored GeneratorExit`.

Expected product behavior: no workflow records a GC/GeneratorExit error, all
modeled pending futures remain alive until explicitly released/cancelled, and
successful branches return the modeled result.

Oracle impact: the interruption ledger, public status/error, or missing weakref
fails if GC destroys the task.

### completed-async-task-reference-leak

Purpose: verify that the strong-reference fix does not keep completed,
cancelled, or errored async workflow tasks alive forever.

Trigger point: after all modeled workflows are terminal and their public
handles/status rows agree with the expected case model.

Injection: poll `dbos._workflow_tasks` after terminal state and force additional
GC cycles to prove done-callback cleanup has run.

Expected product behavior: the task set eventually becomes empty after all
modeled async workflows are terminal.

Oracle impact: a non-empty task set after terminal observation fails as a leak
or cleanup bug, not as the original GC-destruction bug.

## Oracle

The workload should build a task-retention ledger with:

- rung/case/seed and target commit;
- queue name, queue concurrency, workflow IDs, and enqueue order;
- per-workflow suspended-state timestamp and weakref state;
- `dbos._workflow_tasks` snapshots before GC, after GC, after release/cancel,
  and after terminal cleanup;
- `gc.collect()` counts and any captured warnings/log snippets for destroyed
  pending tasks;
- interruption ledger entries including exception type/message from workflow
  `BaseException` handlers;
- public handle result/error and durable workflow status for each workflow.

Required invariant checks:

- every workflow reaches the modeled suspended state before GC assertions start;
- while workflows are suspended, the task pin set and weakrefs prove the
  modeled active workflows remain reachable;
- forced GC does not produce `GeneratorExit`, `RuntimeError: coroutine ignored
  GeneratorExit`, or a premature DBOS `ERROR`;
- released workflows return modeled success values;
- cancelled or application-error branches match the modeled terminal
  classification without GC-destruction signatures;
- after all modeled workflows are terminal, `_workflow_tasks` eventually becomes
  empty.

## Execution / Evidence Notes

Executor cloud run `01KVXR64ZX4DNSCR3Y1FP7FCKM` failed on `case-003` / seed
`7112`: six queued async workflows suspended after normal queue dequeue; three
were cancelled and three were released successfully. Cancelled public handles
raised `DBOSAwaitedWorkflowCancelledError`, durable status was `CANCELLED`, and
the interruption ledger contained no `GeneratorExit` signature.

The task-release invariant failed after terminal observation: `_workflow_tasks`
still contained three live `_execute_workflow_async` tasks. DBOS shutdown then
logged the same three workflow IDs as still active.

## Finding Summary

Status: finding.

Observed finding class: `_workflow_tasks` leak after cancelled queued async
workflows were already terminal in durable public state. The run did not
observe the original `GeneratorExit` destroyed-task signature.

## Regression Notes

Status: pending upstream repair.

Guard: replay `rung-002-queued-async-task-retention-gc-pressure` with the exact
case seeds and require task-reachability, no-GeneratorExit, terminal-state, and
task-release invariants. At minimum, `case-003` / seed `7112` must leave
`_workflow_tasks` empty after cancelled and released workflows reach terminal
state.

## Stale Conditions

Mark stale if DBOS removes queue/dequeue async execution, replaces
`_workflow_tasks` with another task ownership model, changes
`execute_workflow_by_id()` to return task-owning handles for queued async
workflows, changes cancellation terminal-state semantics, or if the workload
can no longer force a suspended async workflow without keeping extra strong
references in the harness.
