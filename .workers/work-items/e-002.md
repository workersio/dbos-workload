# E-002 Stale Queued Recovery Ownership

## Product Promise

A queued workflow dequeued by a dead executor remains owned by its queue during
recovery. A stale recoverer may observe that another recoverer already cleared
queue assignment, but it must not execute the queued workflow body through the
direct recovery path.

## Fault Dimensions

### stale-recovery-snapshot-after-queue-clear

Purpose: exercise the recovery path where two recoverers see the same queued
`PENDING` row snapshot and one recoverer races behind a successful queue
assignment clear.

Trigger point: recoverer B clears the queued row's assignment and returns a
polling handle; stale recoverer C then handles the same captured pending-row
snapshot.

Injection: the workload models a dead dequeued executor by creating a public
queued row with queue listening disabled, moving that durable row to `PENDING`
under `wio-worker-a`, and invoking recovery with duplicated stale pending-row
state.

Reachability evidence: WIO run `01KVVCVG53BKCQZ0MG9S1P1BC6` observed
recoverer B clear assignment and stale recoverer C receive
`clear_queue_assignment=false`.

Expected product behavior: stale recoverer C must not execute queued workflow
body steps outside the queue worker after queue assignment has already been
cleared.

Oracle impact: any ledger body effect under stale recoverer C violates queue
ownership.

Replay metadata: run `01KVVCVG53BKCQZ0MG9S1P1BC6`, case `case-001`, seed
`224`, harness commit `12caea1e77fb2eca04e16d8c919269916b6d56ce`.

## Oracle

The durable row must leave the dead executor state safely, and public handle
results must agree with durable terminal state. The specific finding oracle is
`rung6_no_stale_recoverer_body_effect`: after recoverer B clears queue
assignment, stale recoverer C must not record workflow body ledger effects.
