# E-029 Concurrent Debounce Coalescing (new DELAYED protocol, #752)

## Product Promise

#752 reimplemented the debouncer on `DELAYED` workflows. A debounce enqueues a
DELAYED workflow keyed by `deduplication_id`; each bounce atomically UPDATEs the
DELAYED row (`debounce_delayed_workflow`) to extend `delay_until_epoch_ms`
(capped at `debounce_deadline_epoch_ms`) and replace `inputs`; the key is cleared
on the DELAYEDâ†’ENQUEUED flip. A burst of bounces on one key must coalesce to
exactly one execution per settled quiet-period, carrying the latest committed
input, with no bounce that returned a handle silently dropped.

## Fault Dimensions

### concurrent-bounce-flip-race

Purpose: falsify exactly-once/latest-input coalescing when bounces race each
other and the scheduler's DELAYEDâ†’ENQUEUED transition.

Trigger: M concurrent threads (DBOSClient/`Debouncer`) each issue K
`debounce(key, period, seq)` with monotonically increasing `seq` and small
jitter; period chosen so some bursts land wholly inside one window and some
straddle the flip. The `debounce()` retry loop has a self-documented race
("a same-name debounced holder flipped out of DELAYED mid-bounce â€” retry") and
a loser path that catches `DBOSQueueDeduplicatedError` and re-bounces.

Reachability evidence: `debounce_delayed_workflow` UPDATE requires
`status==DELAYED AND is_debounced`; the flip nulls `deduplication_id` in a
separate transaction (`transition_delayed_workflows`).

Expected product behavior: each settled window runs the target workflow exactly
once with the max `seq` committed in that window; every returning `debounce()`
handle resolves; total executions â‰¤ total bounces; every distinct input set is
either superseded or executed (none silently lost).

Oracle impact: FAIL on a window that double-executes, an execution carrying a
superseded input while a later same-window bounce is lost, a handle that never
resolves, or executions exceeding bounces.
