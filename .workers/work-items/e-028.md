# E-028 GC Two-Phase Orphan → Stale OAOO Replay

## Product Promise

Incremental garbage collection (#751) deletes a workflow's `workflow_status`
(system db) and its `transaction_outputs` (application db). These are two
separate deletes across two databases with no shared transaction. GC must not
leave a workflow's `transaction_outputs` behind after its `workflow_status` is
gone: a later workflow reusing the collected id whose first transaction lands on
the same `function_id` would hit `check_transaction_execution`, which replays
any row matching `(workflow_uuid, function_id)` and skips the step body — so the
new workflow would return the dead workflow's stale output instead of executing.

## Fault Dimensions

### partial-gc-cross-db-orphan

Purpose: verify that a partial GC (system-db delete committed, app-db delete not
run) does not cause a reused workflow id to replay the collected workflow's step
output.

Trigger: run a workflow with one `@DBOS.transaction` step to SUCCESS; call
`dbos._sys_db.garbage_collect(now_ms()+3_600_000, None, 100)` (deletes
`workflow_status`) and deliberately SKIP `dbos._app_db.garbage_collect(...)` —
faithfully modelling the process/connection failure between the two
non-atomic cross-database phases that the PR explicitly designs for ("progress
is preserved if batch fails"). Then start a new workflow with the same
`SetWorkflowID` and a structurally identical transaction.

Reachability evidence: `transaction_outputs` keyed by `(workflow_uuid,
function_id)` (`dbos/_app_db.py`); `check_transaction_execution` replays a
matching row and skips the body; the two GC phases run sequentially in
`dbos/_workflow_commands.py garbage_collect` with no shared transaction.

Expected product behavior: the reused-id workflow executes its transaction
fresh — a new side-effect row in the independent `wio_gc_effects` ledger and a
fresh return value derived from the new input.

Oracle impact: FAIL if the reused-id workflow returns the collected workflow's
stale output, skips the body (no new effect row), or raises a conflict — any of
these is the OAOO/durability violation.

Replay metadata: command shape
`.workers/run-with-postgres.sh .workers/python-runtime.sh .workers/workloads/gc-orphan-oaoo/gc_orphan_oaoo_workload.py --rung rung-001-gc-orphan-oaoo --case <case-id>`.

## Oracle

Independent `wio_gc_effects` ledger (fresh uuid per body execution) compared with
public workflow results and read-only system/app rows:

- `b1_first_output` / `p1_first_output`: first run returns its fresh value.
- `b2_full_gc_clears_both`: full GC removes both `workflow_status` and
  `transaction_outputs` for the id.
- `b3_reused_id_executes_fresh`: after full GC, reused id runs fresh.
- `p2_orphan_window_reached` (reachability): after sys-db-only GC,
  `workflow_status` rows == 0 AND `transaction_outputs` rows == 1.
- `p3_reused_id_after_partial_gc_executes_fresh` (money): reused id after
  partial GC returns fresh output AND records one new effect row.

Do not weaken `p2`/`p3`. A setup failure (Postgres/import) is `SETUP-BLOCK`, not
a product bug.

## Finding Summary

Status: `finding_candidate`. After a partial GC (sys-db `workflow_status`
deleted, app-db `transaction_outputs` orphaned), reusing the collected workflow
id replayed the dead workflow's step output (`result-10`) and skipped the new
body (`effects_n20=0`) — an exactly-once / durable-execution correctness
violation in #751's incremental GC. Reachability invariant `p2` proved the
orphan window is genuine. Upstream filing is held for human triage (not filed
from this run).
