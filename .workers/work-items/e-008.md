# E-008 Debouncer Unrelated Queue Liveness

## Product Promise

Long active debounce windows for hot keys must not consume executor capacity in
a way that prevents trivial workflows on an unrelated DB-backed queue from
completing inside the modeled liveness window.

## Fault Dimensions

### hot-key-debouncer-pressure

Purpose: keep internal hot-key debouncer rows active while measuring unrelated
queue liveness.

Trigger point: after the workload observes an active hot-key internal debouncer
row on `_dbos_internal_queue`, it submits trivial workflows to an unrelated
DB-backed queue.

Injection: use many long debounce windows plus a tiny executor thread pool so
blocked internal debouncer rows can compete with unrelated queue workers.

Reachability evidence: focused cloud run `01KVVJTZVY7JV3JJ231C77DT2E` observed
the hot debouncer row still active immediately before waiting on unrelated
queued handles.

Expected product behavior: unrelated trivial workflows complete inside the
`2.5s` pressure-window bound.

Oracle impact: handle completion around the hot debounce window indicates
unrelated queue starvation.

Replay metadata: full run `01KVVJPJ6YKK2Z2ZN3FCCPNQS2`; focused confirmation
`01KVVJTZVY7JV3JJ231C77DT2E`; workload commit
`9b90b68086da039d3ef8362edde11eca1677c179`.

## Workload Plan

The historical workload uses Postgres and the default build profile. Future
follow-up should first reproduce in WIO deterministic worker or reduce the case
to a standalone DBOS script before filing with a local-reproduction claim.

The Phase 2 prompt trial proposed `rung-007-standalone-repro-minimization` as
the next corridor rung, but executor review rejected it before workload edits:
the spec did not yet define a concrete fault trigger, replay command, case
selector, environment control, or oracle threshold that would distinguish a
real DBOS liveness bug from a WIO deterministic-worker scheduling artifact.
Producer/triage therefore marked it `blocked_workload` rather than leaving a
misleading `ready` rung in the work item.

## Oracle

The workload requires durable evidence that hot debouncer rows are active before
unrelated work is awaited, then measures handle completion for trivial unrelated
queued workflows. Completion above the `2.5s` pressure-window bound violates the
liveness promise only when setup gates and active-row observations pass.

## Execution / Evidence Notes

`10.537618736s`; the full matrix saw the same signature after

so the work item preserves both the finding evidence and the reproduction gap.
