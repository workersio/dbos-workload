# E-033 Portable JSON Input Serializer: Non-Portable + Non-Deterministic

## Product Promise (`portable-inputs-keep-their-types`)

Workflows using the portable JSON serializer receive arguments matching their
Python type hints; datetimes "normalize **deterministically**"; the stored form is
advertised as **portable** JSON; invalid values fail with modeled errors.

## Mechanism (source-grounded, `dbos/_serialization.py`)

`DBOSPortableJSONSerializer.serialize(v)` = `json.dumps(_portableify(v),
separators=(",",":"), ensure_ascii=False)`. Two properties the promise implies but
the implementation breaks:

1. **Portability.** `json.dumps` (no `allow_nan=False`) emits the NON-standard
   tokens `NaN` / `Infinity` / `-Infinity` for `float('nan'/'inf'/'-inf')`.
   RFC-8259 JSON has no such tokens, so a strict / cross-language (JS `JSON.parse`)
   reader REJECTS the stored input. The serializer named `portable_json` is not
   portable for float edges.
2. **Determinism.** `_portableify(set)` = `[_portableify(v) for v in value]` —
   Python set-iteration order, which for STRING elements is randomized by
   `PYTHONHASHSEED` and so varies **between processes**. The same `set` input
   serializes to different bytes in two workers. `frozenset` additionally raises a
   raw `TypeError` (it is not a `set` subclass, so `_portableify` has no branch for
   it) while `set` is accepted — asymmetric type support.

## Verdict

**RED (finding candidate), local — 3 oracles on the real product serializer:**
- **case-001 portability** — `float('nan'/'inf'/'-inf')` serialize to `NaN` /
  `Infinity` / `-Infinity`; all fail a strict RFC-8259 parse. 20 clean values PASS.
- **case-002 determinism** — a `set` / dict-with-set of strings serialized under 4
  distinct `PYTHONHASHSEED`s yields 4 distinct byte strings (`distinct=4`);
  int-sets stay stable (int hashing is not seed-randomized) — the oracle
  discriminates, so its greens are non-vacuous.
- **case-003 end-to-end round-trip** — a real `@DBOS.workflow(serialization_type=
  PORTABLE)` echo persists the input to `dbos.workflow_status.inputs`; for
  `float('nan')` the STORED input is literally
  `{"namedArgs":{},"positionalArgs":[NaN]}` — invalid JSON on disk — **yet the
  workflow reaches terminal SUCCESS** (silent corruption of the durable input, no
  modeled error). Confirms P at the actual storage layer.

Consequence hypothesis (to ground before framing severity for a dossier): any
consumer that re-parses `workflow_status.inputs` with a standard JSON reader, or
compares serialized-input bytes across processes for dedup/idempotency, breaks on
these inputs. Even absent that, both properties violate the stated promise
directly. Correctness class (weight 3).
