# E-031 write_stream From a Step Is Not Exactly-Once

## Product Promise

`DBOS.write_stream` (and `write_stream_async`) is a durable, ordered per-workflow
stream primitive; `DBOS.read_stream` yields each committed value once in offset
order. The framework implements exactly-once for it on the workflow path
(`write_stream_from_workflow` records an `operation_output` and guards
re-execution). A consumer must never see one logical write delivered more than
once — and the public API draws no distinction between writing from a workflow
and writing from a step.

## Mechanism

`write_stream()` (`dbos/_core.py:2170`) routes by caller context:

- **workflow context** → `write_stream_from_workflow` — records an
  `operation_output` and guards re-execution with
  `_check_operation_execution_txn("DBOS.writeStream")`. Exactly-once on replay.
- **step context** → `write_stream_from_step` (`dbos/_sys_db.py:4229`) — inserts
  at `coalesce(max(offset), -1) + 1` with **no** recorded operation and **no**
  guard; retries only on an offset `IntegrityError`.

`streams` is keyed `(workflow_uuid, key, offset)` (`_schemas/system_database.py`)
— `function_id` is stored but excluded from the key. A `@DBOS.step(max_attempts>1)`
re-invokes its body under the **same** `ctx.function_id` on each retry, so a step
that writes a stream value then fails re-inserts a duplicate at a fresh offset on
every attempt. The workflow completes SUCCESS with a silently duplicated stream.

## Freshness / Novelty Check

Not covered by shipped tests: the stream tests exercise workflow-context writes
and read ordering; none drives `write_stream` from a *retrying step*. The
asymmetry with the guarded workflow path is what makes the step-path duplicate a
bug rather than "steps are at-least-once." Sibling class to `e-028` (a
recorded-operation guard present on one of two sibling code paths and omitted on
the other).
