# E-022 Migration Startup Early Exit Under Held Advisory Lock

## Product Promise

DBOS Postgres startup and explicit system database migrations remain bounded
and safe under rolling deploy pressure. If the DBOS schema is already at the
latest migration version, `run_migrations()` must return quickly without
waiting on the migration advisory lock. If the schema is missing, partially
initialized, or behind the latest migration version, startup must not falsely
skip migration work; after the lock holder releases, the schema must reach the
latest version exactly once and support normal DBOS runtime operations.

## Fault Dimensions

### up-to-date-startup-advisory-lock-wait

Purpose: prove the PR `#677` early-exit path avoids acquiring the migration
advisory lock when no migration work is pending.

Trigger point: prepare a fully migrated Postgres system schema, then acquire
advisory lock `1234567890` on a separate connection before calling
`run_migrations()`.

Injection: hold the advisory lock longer than the workload's short early-exit
bound but below the product's 30 second lock timeout.

Expected product behavior: `run_migrations()` returns before lock release,
schema version remains latest, and the lock holder can prove it still owns the
lock at return time.

Oracle impact: elapsed time, lock-held evidence, and unchanged schema version
fail if startup waits on the lock despite an up-to-date schema.

### stale-schema-false-early-exit

Purpose: prove the early-exit predicate does not skip required migrations when
the migration version is behind.

Trigger point: rewind `dbos.dbos_migrations.version` to a modeled prior
version, then hold the advisory lock and start migration.

Injection: release the lock only after the workload observes that the migration
caller has not returned during the hold window.

Expected product behavior: migration does not return while the lock is held,
then reaches the latest version after release.

Oracle impact: returning before lock release or leaving the final version stale
fails as a safety bug.

### partial-schema-migration-repair

Purpose: prove a partially initialized schema, especially a missing
`dbos_migrations` table, is not mistaken for up-to-date state.

Trigger point: create the DBOS schema but drop or omit the migrations table,
then hold the advisory lock and start migration.

Injection: release the lock after the blocked window and verify the migration
runner repairs the schema.

Expected product behavior: migration creates/repairs `dbos_migrations`, reaches
latest version, leaves required system tables queryable, and passes a minimal
runtime smoke.

Oracle impact: missing table, wrong version, duplicate rows, or failed runtime
smoke fails the case.

## Oracle

The workload should build a migration liveness ledger with:

- rung/case/seed and target commit;
- generated database or schema names and cleanup ownership;
- latest migration count and the version row before mutation;
- advisory-lock holder connection ID, acquisition time, release time, and lock
  still-held checks;
- per-worker start/end timestamps, elapsed time, exception, and whether it
  returned before or after lock release;
- final `dbos_migrations` rows and required DBOS system table presence;
- minimal DBOS runtime smoke after migrated state is expected usable.

Required invariant checks:

- up-to-date migration calls return while the external advisory lock is still
  held and below the modeled early-exit bound;
- stale or partial schema migration calls do not return before lock release;
- after release, stale or partial schemas reach the latest version within a
  healthy bound;
- final `dbos_migrations` has exactly one version row equal to latest;
- required system tables exist after migration repair;
- runtime smoke can create and run a trivial workflow or equivalent DBOS
  operation after the schema is claimed current;
- no worker reports migration-lock timeout for up-to-date cases.

## Finding Summary

Status: finding candidate.

Observed finding class: many concurrent up-to-date warm starts waited behind a
held migration advisory lock instead of returning while the lock remained held.
In the focused replay, six workers started while the lock was held and returned
after approximately `14.34s` to `16.28s`; the advisory lock had been released
after roughly `8.89s`. The full matrix reproduced the same case with worker
elapsed times of approximately `14.57s` to `16.00s`.

Not observed in this run: stale schema false early exit, partial schema repair
failure, wrong final migration version, duplicate final migration rows, or
runtime smoke failure.
