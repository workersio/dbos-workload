# E-017 Adopted Event Loop Shutdown Liveness

## Product Promise

DBOS shutdown and background-event-loop helpers must not deadlock when DBOS has
adopted an application event loop and timeout waiter tasks are still pending.
`DBOS.destroy()` must complete within a bounded window, clear or cancel timeout
tasks, and leave the runtime reusable. Blocking coroutine submission from the
target loop's own thread must fail loudly instead of hanging forever.

## Fault Dimensions

### adopted-loop-timeout-destroy-deadlock

Purpose: verify that `DBOS.destroy()` does not block forever when called from
the same event-loop thread that owns pending DBOS timeout waiter tasks.

Trigger point: after DBOS has launched from inside a running application event
loop, after a workflow under `SetWorkflowTimeout` has completed, and after the
workload observes a non-empty `dbos._timeout_tasks` set.

Injection: call `DBOS.destroy(destroy_registry=True)` from the adopted loop
thread, and separately call blocking `BackgroundEventLoop.submit_coroutine()`
from its own target loop to exercise the loud-failure guard.

Reachability evidence: PR `#722` added target tests that reproduce the
self-deadlock with a dedicated event-loop thread and a pending timeout task.
Target `dbos/_dbos.py` now cancels timeout tasks directly when the current loop
is the target loop.

Expected product behavior: destroy returns before the bounded join timeout,
DBOS-owned timeout tasks are cancelled/cleared, same-loop blocking submission
raises a deadlock-class `RuntimeError`, and a fresh DBOS instance can relaunch.

Oracle impact: the lifecycle ledger fails if shutdown hangs, same-loop blocking
submission hangs or succeeds, timeout tasks leak, or relaunch is contaminated by
the previous runtime instance.

Replay metadata: expected command shape
`python .workers/workloads/runtime-shutdown-event-loop-liveness/runtime_shutdown_event_loop_liveness_workload.py --rung rung-001-adopted-loop-timeout-destroy-liveness --case <case-id> --seed <seed>`.

## Oracle

The workload should build a lifecycle ledger with:

- loop identity and DBOS target loop identity;
- worker thread name and join duration;
- workflow ID and timeout seconds;
- `dbos._timeout_tasks` count before destroy and after teardown;
- destroy start/end timestamps and exception outcome;
- same-loop `submit_coroutine` result/error classification;
- relaunch workflow result/status.

Required invariant checks:

- DBOS adopted the running event loop before the destroy case;
- a DBOS-owned timeout task exists before destroy;
- destroy returns within the modeled bound and leaves no live worker thread;
- timeout tasks are cleared or cancelled after destroy;
- same-loop `submit_coroutine()` raises a deadlock-class `RuntimeError`
  promptly;
- relaunch after destroy succeeds and a trivial timeout workflow returns the
  modeled result.
