# E-018 Concurrent Schedule Apply Live Update

## Product Promise

Public schedule application is atomic, idempotent, and live-update safe under
concurrent callers. Re-applying the same `schedule_name` must leave exactly one
durable schedule row, preserve operator-controlled state such as pause status
and `last_fired_at` where the API contract requires it, replace mutable fields
with the latest intended definition, restart the live scheduler thread when the
definition changes, and avoid duplicate live executions or stale contexts.

## Fault Dimensions

### concurrent-apply-schedule-race

Purpose: prove public schedule application is a single logical registry update
even when several workers publish the same logical schedule at the same time.

Trigger point: after DBOS is launched and the scheduled workflow is registered,
release multiple worker threads through a barrier so they call
`DBOS.apply_schedules()` or `DBOSClient.apply_schedules()` for the same
`schedule_name` concurrently.

Injection: vary caller surface (`DBOS` class API versus `DBOSClient`), worker
count, and schedule definition details while keeping a single modeled logical
schedule name.

Reachability evidence: the pinned target uses delete-then-create in both public
apply paths; PR `#741` replaced that with an upsert and added a concurrent
product test that expected no caller exceptions and exactly one row.

Expected product behavior: no caller raises, public/client reads expose exactly
one schedule row, and the durable row matches the latest modeled definition.

Oracle impact: the workload fails on any unique-constraint, missing-row, stale
field, duplicate-row, or swallowed worker-thread exception evidence.

Replay metadata: expected command shape
`.workers/run-with-postgres.sh .workers/python-runtime.sh .workers/workloads/schedule-registry-concurrency/schedule_registry_concurrency_workload.py --rung rung-001-concurrent-apply-live-update --case <case-id> --seed <seed>`.

### live-schedule-reapply-stale-thread

Purpose: prove a changed schedule definition takes effect in the live dynamic
scheduler without requiring process restart and without duplicate old-context
ticks.

Trigger point: after an every-second schedule has fired at least once with
context version 1, reapply the same `schedule_name` with context version 2.

Injection: reapply while the scheduler loop is active, then observe live
workflow executions across a bounded grace window.

Reachability evidence: target `_scheduler.dynamic_scheduler_loop()` keys live
threads by `schedule_id`; PR `#741` changed upsert conflict handling to replace
`schedule_id` so the old thread is stopped and a new one picks up changed
fields.

Expected product behavior: the reapply produces a fresh schedule id, subsequent
live ticks use the latest context after the grace window, workflow IDs remain
unique, and cleanup quiesces the schedule.

Oracle impact: the fired-context ledger fails if the old context continues,
duplicate workflow IDs appear, the schedule id does not change after live
definition update, or cleanup does not stop new ledger entries.

## Oracle

The workload should build a schedule-registry ledger with:

- rung/case/seed and target commit;
- public/client caller definitions and barrier timing;
- per-caller outcome, exception class/message, and elapsed time;
- schedule snapshots before apply, after concurrent apply, after reapply, and
  after cleanup;
- expected model state for schedule id, schedule name, cron, context,
  `status`, `last_fired_at`, `cron_timezone`, and `queue_name`;
- fired workflow ledger with scheduled timestamp, workflow id, context version,
  start timestamp, and terminal status;
- cleanup proof that the schedule row was deleted and no modeled live thread
  can keep adding ledger entries after cleanup.

Required invariant checks:

- no public or client apply caller raises a unique-constraint, missing-row,
  serialization, or schedule conflict error;
- after each apply phase, `list_schedules()` and `get_schedule()` expose
  exactly one row for the modeled `schedule_name`;
- durable mutable fields match the latest modeled definition;
- modeled operator state is not accidentally reset; paused status remains
  paused across reapply unless the public contract intentionally says apply
  should reactivate, and `last_fired_at` is not cleared after a live schedule
  has fired;
- changed live definitions produce a fresh `schedule_id` so the dynamic
  scheduler replaces the old thread;
- after the reapply grace window, fired ledger entries for the schedule use only
  the latest context version;
- workflow IDs for live ticks are unique for modeled scheduled timestamps, and
  no duplicate ledger effect appears for the same workflow ID.

## Execution / Evidence Notes

Executor cloud run `01KVXQEDH5S05FE916ES6GPT96` failed on
`case-001` / seed `7410`: eight public `DBOS.apply_schedules()` callers
concurrently applied `wio-sched-case-001-7410`; one caller succeeded and seven
raised `DBOSException` with message
`DBOS Error: Schedule 'wio-sched-case-001-7410' already exists`.

The final durable row was present and matched the model, so the finding is
caller-visible non-idempotence during concurrent apply rather than an absent or
duplicated final row.
