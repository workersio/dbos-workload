# Run E-029 — Concurrent debounce coalescing (#752 DELAYED protocol)

## Verdict

**GREEN — NOT a finding. Cloud-confirmed** (run `01KX47FF2KHTYPY50VCVFSP6BX`).
The new (#752) DELAYED-based debouncer behaved correctly under concurrent bounce
bursts: every window ran its body exactly once, carried its window-latest input,
dropped no handle, exceeded neither the bounce count nor leaked cross-window
inputs. A first cloud run surfaced a **miscalibrated oracle** (a single-window
timing assumption that does not hold on the slower cloud VM), not a product bug;
the oracle was recalibrated and the re-run is green under even heavier cloud
fragmentation (8 windows).

**Root cause = workload, not product.** On the cloud VM the first bounce
(seq=1) created a DELAYED workflow whose delay expired and flipped to ENQUEUED
before the remaining bounces arrived, so the burst formed TWO debounce windows
(seq=1 alone, then seq 2–9 → seq=9). Each window ran exactly once with its
window-latest input — correct debouncer behavior. The `exactly_one_execution`
invariant encoded a timing assumption (all 9 bounces coalesce into one window)
that only held on the fast local box. Classified per the overview reality note:
"classify with run command and invariant evidence, not labels alone." This is
**not** a maintainer-reportable bug and was not filed.

## Recalibration

Oracle fixed to be cloud-robust without weakening the falsification surface:
- Strong single-window assertions (`exactly_one_execution`, `single_window_max_seq`)
  are now CONDITIONAL — asserted only when exec_count==1; a split into multiple
  windows takes the bounded per-window path instead of failing.
- Always-on product invariants stay strict and keep the RED surface live:
  `no_double_exec`, `exec_le_bounces`, `all_handles_resolved`,
  `latest_input_wins` (global max present), per-window latest-wins.
- Single-window cases use a larger period so the burst comfortably fits one
  window even on a slow VM, raising the chance the strong path is exercised.
