## Classification

Finding.

Case `case-001` enqueued four gated async workflows on partition
`partition-a-2431` and four on `partition-b-2431`. The release gate remained
closed, so every ledger start represented live simultaneous execution, not
terminal-count inference. Before any release, partition `partition-a-2431`
reached three active workflows even though the queue was partitioned and
registered with `worker_concurrency=2`.

The target code path supports the failure mechanism without expanding the claim
beyond observed evidence. The queue worker computes per-partition capacity from
`dbos._active_workflows_set.count_for_queue(queue.name, key)` before calling
`start_queued_workflows(...)`. For async dequeued workflows,
`execute_workflow_by_id(...)` schedules execution through the background event
loop, and `_execute_workflow_async(...)` acquires the active-set bucket only when
the async task starts. The confirmed product symptom is that this path allowed
three same-partition async workflows to become active under a modeled cap of
two.
