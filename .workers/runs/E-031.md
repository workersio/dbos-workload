# Run E-031 — write_stream from a step is not exactly-once

## Verdict

**RED (finding candidate).** Confirmed on-box (local Postgres). `DBOS.write_stream`
issued from inside a retrying `@DBOS.step` re-inserts the same logical value on
every retry attempt, so a single logical write appears in the stream K times. The
identical write from a workflow context appears exactly once. Same public API, no
documented distinction → an exactly-once / durable-execution correctness
violation. The workflow still completes SUCCESS (the duplication is silent).

## Source grounding

- `dbos/_core.py:2170 write_stream` — routes to `write_stream_from_workflow`
  (workflow ctx) or `write_stream_from_step` (step ctx).
- `dbos/_sys_db.py:4229 write_stream_from_step` — inserts at `max(offset)+1`, no
  `_check_operation_execution_txn` guard, no recorded operation. Its workflow
  sibling (`:4265`) has the guard.
- `dbos/_schemas/system_database.py streams` — `PrimaryKeyConstraint(
  workflow_uuid, key, offset)`, excludes `function_id`.
- `dbos/_core.py` step retry loop — retries re-run the body under the same
  `ctx.function_id`.

## v0.6.0 oracle plane

- **primary** — `stream_exactly_once_<case>`: observed copies of the single acked
  value must == 1.
- **durawatch** — acked stream content (`[payload-v1]`) re-observed across the
  `[0s, 2s, 5s]` ladder; a duplicate is a persistent mutation of the acked
  observable → FAIL (also guards against loss).
- **crashclock** — retry multiplicity K is a declared `op_index` space
  (`retry_multiplicity` ∈ [2,4]), seed-derived; a `CLOCK` line records the point.
  VOID if K<2 (retry never armed).
- **liveness** — watchdog thread fails the run if a workflow hangs (120s budget).
- **terminal sweep** — every workflow driven must reach terminal SUCCESS.
- **selftest** — `ORACLE_SELFTEST` plants a duplicate into the control so the
  oracle must go RED.
