# Run E-032 — DBOS.send from a step is not exactly-once

## Verdict

**RED (finding candidate) — CLOUD-CONFIRMED.** A single `DBOS.send` issued from
inside a retrying `@DBOS.step` is delivered once per attempt; the same send from a
workflow context is delivered once. Confirmed on local Postgres against BOTH
released `dbos==2.26.0` and current `main` (fork HEAD), AND on WIO cloud with the
full v0.6.0 oracle-plane workload. Novel, unfixed on main — the highest-value new
finding of the standing-pool scout pass.

## Differential (recipient-observable)

| sender | recipient received | copies |
|--------|--------------------|:------:|
| `DBOS.send` inside `@DBOS.step(max_attempts=3)` (fails twice) | `['order-1','order-1','order-1']` | 3 (RED) |
| `DBOS.send` from the workflow body (control) | `['order-1']` | 1 (GREEN) |

## Source grounding

- `dbos/_core.py:2079` `send_bulk` dispatch — `if cur_ctx.is_workflow()` records a
  guarded step; the `else` (step OR client) passes `workflow_id=None,
  function_id=None`.
- `dbos/_sys_db.py:2818/2890` `_send_bulk_txn` — OAOO block gated on
  `workflow_id is not None`, skipped for the step path.
- `dbos/_sys_db.py:2852` — fresh `message_uuid` per call; `:2876`
  `on_conflict_do_nothing` never dedups without an idempotency key.

## Depth / seed rationale

Deterministic; not seed-sensitive. The duplication is a structural code-path gap —
a plain `max_attempts=3` step retry reproduces it every run (3 copies), with no
crash, timing window, or seed sweep. The workflow-context control proves the setup
is sound, so a green control + red treatment cannot be an environment artifact.
