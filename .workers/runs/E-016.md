## Review

Verdict: `KEEP`

The workload adds a sys-db retry/idempotence failure surface that existing
message, serializer, async-checkpoint, datasource, and recovery workloads did
not cover. It would fail if replay consumed a second message, if timeout replay
consumed a late message, if same-child retry conflicted, if different-child
retry failed to conflict, if empty child ids persisted rows, or if
`record_get_result` advanced the parent function id twice under retry. The
main residual risk is that receive and child-edge cases use explicit
committed-state replay surrogates rather than an engine fault injected after
commit; `case-004` uses direct scoped retry injection.
