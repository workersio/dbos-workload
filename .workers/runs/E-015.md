- Finding:
  - A SQL-origin queued workflow whose stored `authenticated_roles` do not
    satisfy `@DBOS.required_roles(["admin"])` is dequeued and left `PENDING`
    indefinitely instead of being finalized as an authorization `ERROR`.
  - The failure is not a setup failure: DBOS launched, migrations completed, the
    queue was registered/listening, SQL enqueue inserted the modeled auth row,
    and the allowed-role control case completed on the same prepared image.
  - Source-path check supports the failure mode. `start_queued_workflows` marks
    selected rows `PENDING` before execution. The queue worker catches
    `execute_workflow_by_id` exceptions only for logging. In `_execute_workflow_wthread`,
    `check_required_roles` runs before `_get_wf_invoke_func`, which is the path
    that persists workflow success/error outcomes. A denied role can therefore
    escape without `update_workflow_outcome(ERROR)`, leaving the dequeued row
    stuck as `PENDING`.
