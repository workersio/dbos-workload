# Run E-028 — GC two-phase orphan → stale OAOO replay

## Verdict

**RED (finding candidate).** Confirmed on WIO cloud. A partial garbage collection
(system-db `workflow_status` deleted, application-db `transaction_outputs` not yet
deleted) orphans a completed workflow's transaction output. A new workflow that
reuses the collected workflow id replays the dead workflow's recorded step output
and skips its own body — an exactly-once / durable-execution correctness violation.

## Mechanism

`dbos/_workflow_commands.py:garbage_collect` runs two phases with no shared
transaction across two databases: (1) `dbos._sys_db.garbage_collect` deletes
`workflow_status` (system db, batched watermark loop, one committed txn per
batch), then (2) `dbos._app_db.garbage_collect` deletes `transaction_outputs`
(application db). A process/connection failure between the phases — a state the
PR explicitly designs for ("progress is preserved if batch fails") — leaves
`transaction_outputs` orphaned. `transaction_outputs` is keyed by
`(workflow_uuid, function_id)`; `check_transaction_execution` (`dbos/_app_db.py`)
returns any matching row and skips the step body. A workflow reusing the
collected id (a normal idempotency-key pattern) whose first transaction lands on
the same `function_id` collides with the orphan and replays the dead output.

## Injection faithfulness

The workload drives the real system-db GC phase (`dbos._sys_db.garbage_collect`)
and deliberately does not run the app-db phase, reproducing the exact durable
state a crash-between-phases produces. This is state-modeling of an
acknowledged partial-failure path, not a mocked state: `p2` reads the real
orphan row from Postgres. A second reachability path (app-db's own batched loop
failing partway — the PR's resumable test only injected failure on the sys-db
side) produces the same orphan and is a natural deeper rung.

## Strengthening: case-003 app-db-batch partial failure (public-API trigger)

Cloud run `01KX4BZJHVB2V4MKPA9FDY08JE` (commit `f834259`), state `failed`.
A more file-defensible trigger for the SAME orphan → stale-replay, through the
**fully public** `garbage_collect(dbos, cutoff, None, batch_size=1)`:
- `a1_first_output` PASS — target ran, `result-30`.
- `a2_appdb_batch_partial_orphan` PASS — the public GC ran the sys-db phase
  fully, then entered the app-db batched delete loop; a single transient fault
  was injected into that loop (`fault_fired=true`, `appdb_begin_calls=2`,
  `fillers_deleted=1` — batch 1 committed, batch 2's `begin()` raised). Target's
  `workflow_status`=0, `transaction_outputs`=1 → orphaned by the loop failure,
  not by skipping the phase.
- `a3_reused_id_after_appdb_partial_executes_fresh` **FAIL** — reuse returned
  stale `result-30`, `effects_n40=0` (body skipped).

Why it matters: this removes case-002's white-box "skip the whole app-db phase"
objection. The orphan is produced by the app-db batched loop failing partway —
exactly the partial failure #751 designs for ("progress is preserved if batch
fails") and whose resumable test only faults the **sys-db** side. The sys-db
engine is untouched here; only the app-db loop takes one transient fault.
